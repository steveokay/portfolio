## Problem

A user-facing API endpoint aggregating data across three Postgres tables was degrading under load. p99 latency spiked to 340ms during peak (US East, ~50k req/s), causing frontend timeouts and a 0.4% error rate on checkout flows.

## Constraints

- Zero downtime — endpoint served checkout
- Postgres 14, no schema changes allowed mid-quarter
- Fix had to be operable by on-call engineers unfamiliar with the codebase

## Architecture decision

Profiled with `pprof` under production load replay. Found three issues: (1) ORM generating N+1 queries per request, (2) JSON serialization creating GC pressure at 50k req/s, (3) no caching for near-static reference data called on every request.

Rewrote the query as a single JOIN, switched to `encoding/json` streaming encoder, added Redis with TTL=60s for reference data.

## Trade-off not taken

A read replica with pgBouncer would have been architecturally cleaner and future-proof, but added 2 weeks of runway and on-call complexity. The cache + query rewrite shipped in 3 days with equivalent p99 outcomes.
